# RL-Based Intelligent API Rate Limiting

## Purpose

This project simulates an API gateway that must decide how aggressively to admit traffic under changing load. The goal is to maximize useful throughput while controlling queueing, latency, and dropped requests, with extra protection for high-priority traffic.

The system uses reinforcement learning to learn a rate-limiting policy. A PPO agent interacts with a discrete-time traffic simulator and learns when to decrease, hold, or increase the admission limit.

## High-Level Architecture

The system has five main parts:

1. `api_rate_limiter_rl/env.py`
   Discrete-time environment for traffic generation, admission control, retries, queueing, service, and reward calculation.
2. `api_rate_limiter_rl/ppo.py`
   PPO actor-critic implementation used to learn the control policy.
3. `api_rate_limiter_rl/baselines.py`
   Rule-based policies used for comparison against PPO.
4. `api_rate_limiter_rl/experiments.py`
   Training loop, evaluation loop, plotting, JSON export, and report generation.
5. `api_rate_limiter_rl/config.py`
   Central configuration for scenarios, simulator parameters, and PPO hyperparameters.

## Core Idea

At every simulator time step:

1. Traffic arrives with different priorities.
2. The controller picks one of three actions:
   - decrease rate limit
   - hold rate limit
   - increase rate limit
3. Requests above the current rate limit are rejected.
4. Some rejected requests retry later.
5. Accepted requests enter a queue.
6. The service layer processes requests with priority-aware scheduling.
7. The environment computes reward from throughput, latency, drops, high-priority drops, and overload.

The PPO agent learns a policy that maps current system state to one of the three actions.

## Traffic Model

Traffic is generated by scenario definitions in `api_rate_limiter_rl/config.py`.

Supported traffic patterns:

- `low_load`
  Light steady traffic.
- `high_load`
  Sustained heavy traffic.
- `bursty`
  Moderate traffic with periodic bursts.
- `oscillating`
  Load that rises and falls over time.
- `training_mix`
  Generic training placeholder. Real training episodes sample from a weighted mix of the main scenarios.

Each scenario controls:

- base arrival rate
- oscillation amplitude and period
- burst interval, burst length, and burst multiplier
- probability of low, medium, and high priority requests

Incoming request counts are sampled stochastically. Priority labels are also sampled stochastically.

## Environment Dynamics

The environment class is `RateLimitEnv`.

### Admission control

The environment maintains a current `rate_limit`. The chosen action changes that limit in fixed steps within configured bounds:

- min rate limit: `2`
- max rate limit: `18`
- step size: `2`

If more requests arrive than the current rate limit allows, the excess is rejected immediately.

### Retries

Rejected requests may retry after a random delay. Retry probability depends on request priority, so some priorities are more persistent than others. This makes the traffic process more realistic because overload can spill into future timesteps instead of disappearing instantly.

### Queueing and service

Accepted requests go into a queue. The service layer has a fixed capacity per time step:

- service capacity: `12`

If the queue grows too large, a hard cap prevents unbounded growth. Service is priority-aware:

- high-priority requests are served first
- then medium-priority
- then low-priority

Within the same priority, older requests are served first.

### Latency model

Latency depends on:

- how long a request waited
- current queue pressure

This means the controller is punished not just for dropping traffic, but also for allowing queues to build up.

## State Representation

The PPO agent observes a compact numeric state vector from `RateLimitEnv._state(...)`.

State includes:

- normalized incoming traffic rate
- normalized queue length
- rolling drop rate
- normalized retry pressure
- normalized current rate limit
- distribution of current arrivals across priorities
- distribution of queued requests across priorities

This state gives the agent visibility into both immediate load and congestion trend.

## Action Space

The action space is discrete with three actions:

- `0`: decrease rate limit
- `1`: hold current rate limit
- `2`: increase rate limit

The agent does not directly pick an absolute rate limit. It performs local control adjustments. That keeps the action space small and makes training more stable.

## Reward Function

Reward is designed to reflect rate-limiter quality, not just raw throughput.

Positive pressure:

- higher throughput

Negative pressure:

- higher latency
- higher total drop rate
- higher high-priority drop rate
- overload beyond queue threshold

Configured reward weights from `EnvConfig`:

- throughput weight: `1.1`
- latency weight: `1.7`
- drop weight: `1.2`
- high-priority drop weight: `3.1`
- overload weight: `1.0`

Important design choice:

High-priority drops are penalized much more heavily than generic drops. This pushes the learned policy toward graceful degradation under stress.

## PPO Learning System

The learning code lives in `api_rate_limiter_rl/ppo.py`.

### Model

The PPO policy is an actor-critic neural network:

- shared MLP trunk
- actor head for action logits
- critic head for state value

Current default hidden size:

- `48`

### Training method

The PPO agent uses:

- clipped policy objective
- generalized advantage estimation (GAE)
- minibatch updates
- entropy bonus
- gradient clipping

Default hyperparameters from `PPOConfig`:

- actor learning rate: `0.005`
- critic learning rate: `0.008`
- gamma: `0.97`
- lambda: `0.93`
- clip epsilon: `0.2`
- update epochs: `8`
- minibatch size: `128`
- training episodes: `260`

### Device selection

`PPOConfig.device = "auto"` chooses accelerated compute when available. In your current run, PyTorch detected Apple Metal (`MPS`), so training can use Apple GPU acceleration.

## Training Flow

Training is orchestrated by `train_ppo(...)` in `api_rate_limiter_rl/experiments.py`.

For each episode:

1. Sample a training scenario from a weighted mix.
2. Run one rollout episode in the simulator.
3. Collect states, actions, rewards, log-probabilities, and value estimates.
4. Convert the rollout into a PPO batch with GAE returns.
5. Update the actor-critic network.
6. Record episode metrics for plotting and reporting.

The weighted sampling favors stressful conditions more than a uniform sampler. That makes the controller train more often on cases where rate limiting actually matters.

## Baselines

The project compares PPO against several hand-built policies:

- `fixed`
  Uses a fixed admission threshold.
- `token_bucket`
  Standard token-bucket style control.
- `adaptive`
  Queue-threshold heuristic that expands or contracts limit based on congestion.
- `priority_heuristic`
  Rule-based policy aware of priority behavior.

These baselines matter because PPO should beat or at least compete with practical heuristics, not just trivial controls.

## Evaluation Flow

After training, `run_experiments(...)` evaluates PPO and all baselines on:

- `low_load`
- `high_load`
- `bursty`
- `oscillating`

Each policy is run for multiple evaluation episodes, then aggregated by mean and standard deviation.

Tracked metrics include:

- throughput
- latency
- drop rate
- high-priority success rate
- total reward

Plots are generated per scenario for headline metrics.

## Generated Outputs

Running `python run_experiments.py` produces:

- `artifacts/REPORT.md`
  Human-readable experiment summary.
- `artifacts/training_history.json`
  Per-episode training metrics.
- `artifacts/evaluation_results.json`
  Full aggregated evaluation metrics.
- `artifacts/reward_summary.json`
  Compact scenario reward comparison.
- `artifacts/plots/`
  Training curves and per-scenario comparison plots.

## What This System Demonstrates

This system is not a production API gateway. It is a research-style simulator for studying adaptive rate limiting under uncertainty.

What it demonstrates well:

- adaptive control under changing traffic
- tradeoff between throughput and latency
- explicit protection of high-priority traffic
- comparison between learned and heuristic policies

What it abstracts away:

- real network delays
- distributed coordination
- multi-instance gateway state sharing
- request-specific resource costs
- SLA enforcement beyond the defined reward function

## Short Explanation For Presentation Use

This project builds a simulated API rate limiter that learns to adapt its admission threshold using reinforcement learning. The simulator generates varying traffic patterns, assigns request priorities, models retries and queueing, and measures throughput, latency, and drops. A PPO agent observes the system state and decides whether to tighten or relax the rate limit. The learned policy is then compared with classical heuristics such as fixed thresholds and token buckets across multiple traffic scenarios.
